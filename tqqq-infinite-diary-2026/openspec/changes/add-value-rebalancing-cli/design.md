# 기술 설계 문서

## 컨텍스트 (Context)

### 배경
사용자가 레버리지 ETF(TQQQ)에 투자할 때 "라오어 밸류 리밸런싱" 전략을 사용하여 체계적으로 매수/매도 시점을 결정하고자 합니다. 현재는 수동으로 계산하고 있어 오류 가능성이 높고 효율性이 낮습니다.

### 제약사항
- Python 3.8+ 환경에서 실행 가능해야 함
- 인터넷 연결이 없을 때도 오프라인으로 동작해야 함 (마지막 가격 데이터 사용)
- Windows 10+ 및 Unix 계열 OS 모두 지원
- 외부 서버 없이 로컬에서만 실행 (데이터 프라이버시)
- exchange-rate-tracker와 유사한 UI 패턴 따르기

### 이해관계자
- **주 사용자**: TQQQ 투자자
- **개발자**: Claude (AI 코딩 어시스턴트)

## 목표 / 비목표 (Goals / Non-Goals)

### 목표 (Goals)
1. 정확한 밸류 리밸런싱 계산 자동화
2. 2주 단위 V값 추적 및 리밸런싱 신호 제공
3. 5주 분할 매수 시뮬레이션으로 의사결정 지원
4. 직관적인 CLI 사용자 인터페이스
5. 영구적인 데이터 저장 및 관리

### 비목표 (Non-Goals)
1. 실시간 가격 알림 (일 배치 작업으로 충분)
2. 웹/모바일 UI (CLI만 제공)
3. 다른 종목 지원 (TQQQ만)
4. 복잡한 백테스팅 기능
5. 자동 매매 실행 (시그널만 제공)

## 기술적 결정 (Decisions)

### 1. 프로그래밍 언어: Python 3.8+
**결정**: Python 사용

**이유**:
- 금융 데이터 처리에 최적의 라이브러리 생태계 (yfinance, pandas)
- CLI 개발에 용이한 argparse, prompt_toolkit 등
- 데이터 계산 로직 구현이 간결
- 교차 플랫폼 지원 우수

**대안 고려**:
- TypeScript/Node.js: 금융 라이브러리가 Python만큼 풍부하지 않음
- Bash script: 복잡한 계산 로직 구현 어려움

### 2. 가격 데이터 소스: 야후 파이낸스 API (yfinance)
**결정**: yfinance 라이브러리 사용

**이유**:
- 무료이며 안정적
- TQQQ 과거 데이터 제공
- Python에서 쉽게 사용 가능
- 커뮤니티 지원 활발

**대안 고려**:
- Alpha Vantage: API 키 필요, 무료 제한 있음
- 직접 스크래핑: 유지보수 부담

### 3. 데이터 저장: CSV 파일
**결정**: CSV 파일 사용

**이유**:
- 단순한 데이터 구조 (테이블 형태)
- 휴대성이 좋고 사람이 읽기 쉬움
- git으로 버전 관리 가능
- exchange-rate-tracker와 일관성

**데이터 구조**:

**리밸런싱 기록 (records.csv)**:
```csv
date,price,shares,pool,v1,value2,signal,created_at
2026-03-14,45.32,100,50000,100000,105000,HOLD,2026-03-14T10:30:00
```

**가격 캐시 (price_cache.csv)**:
```csv
date,close_price,fetched_at,source
2026-03-14,45.32,2026-03-14T10:30:00,API
2026-03-13,44.50,2026-03-13T10:30:00,API
2026-03-12,46.10,2026-03-12T10:30:00,Manual
```

**초기 포트폴리오 (initial_portfolio.csv)**:
```csv
created_at,initial_cash,initial_shares,average_price
2026-03-01,100000,50,40.50
```

**대안 고려**:
- SQLite: 복잡성 증가, 단순查询에 과잉
- JSON: 구조화된 데이터에는 좋으나 테이블 데이터에는 CSV가 적합

### 4. CLI 라이브러리: argparse + colorama
**결정**: 표준 라이브러리 argparse + colorama 조합

**이유**:
- argparse: 표준 라이브러리, 가벼움
- colorama: 교차 플랫폼 색상 지원 (Windows 포함)
- exchange-rate-tracker와 패턴 일치

**대안 고려**:
- prompt_toolkit: 기능은 강력하나 과잉일 수 있음
- click: 복잡한 CLI에 적합하나 단순한 메뉴 기반에는 불필요

### 5. 아키텍처: 모듈형 구조
**결정**: 기능별 모듈 분리

**모듈 구조**:
```
value_rebalancing/
├── __init__.py
├── main.py              # CLI 진입점, 메뉴 처리
├── calculator.py        # 밸류 계산 로직
├── data_manager.py      # 리밸런싱 기록 CRUD 연산
├── price_fetcher.py     # API 가격 조회, 캐싱
├── price_cache_manager.py  # 가격 캐시 CRUD 연산
├── simulator.py         # 5주 분할 매수 시뮬레이션
└── models/
    ├── record.py        # 리밸런싱 기록 모델
    └── price_cache.py   # 가격 캐시 모델
```

**이유**:
- 단일 책임 원칙 (SRP) 준수
- 테스트 용이성
- 유지보수성 향상

## 리스크 / 트레이드오프 (Risks / Trade-offs)

### 리스크 및 완화

| 리스크 | 영향 | 확률 | 완화 방안 |
|--------|------|------|----------|
| 야후 파이낸스 API 장애 | 높음 | 중간 | 마지막 가격 캐싱, 수동 입력 대비책 |
| 계산 공식 오류 | 높음 | 낮음 | 단위 테스트, 사용자 검증 기능 |
| CSV 데이터 손상 | 중간 | 낮음 | 정기 백업, 검증 로직 |
| Windows 호환성 문제 | 중간 | 낮음 | colorama 사용, 테스트 |

### 트레이드오프

1. **자동화 vs 유연성**
   - 선택: 자동화를 기본으로 하되 수동 입력 지원
   - 이유: 사용자가 API 대신 원하는 가격 입력 가능

2. **단순성 vs 기능**
   - 선택: 핵심 기능에 집중, 백테스팅 등 복잡한 기능 제외
   - 이유: 빠른 개발과 유지보수성

3. **정확성 vs 성능**
   - 선택: 계산 정확성 최우선, API 성능은 부차적
   - 이유: 재무 계산에서 정확성이 핵심

## 개방형 질문 (Open Questions)

1. **가격 데이터 주기**: 현재는 2주마다 금요일 종가 기준인데, 이를 자동으로 감지할지 사용자가 명시할까요?
   - **제안**: 사용자가 "계산 실행" 시 자동으로 가장 최근 금요일 종가 사용

2. **시뮬레이션 가격 시나리오**: 5주간 어떤 가격 변동 폭을 가정할까요?
   - **제안**: ±10%, ±20%, ±30% 세 가지 시나리오 제공

3. **초기 V값 설정**: 첫 번째 계산 시 v1 값을 어떻게 설정할까요?
   - **제안**: 사용자 입력 또는 E(평가금액)로 초기화

4. **리밸런싱 실행**: 시그널만 제공할지 실제 매수/매도 금액도 계산할까요?
   - **제안**: Phase 1에서는 시그널만, Phase 2에서는 금액 계산 고려
