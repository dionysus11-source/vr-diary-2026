# value-rebalancing-cli Specification

## Purpose
라오어 밸류 리밸런싱 전략을 사용하여 TQQQ 레버리지 ETF 투자의 매수/매도 시점을 체계적으로 관리하는 CLI 프로그램

## ADDED Requirements

### Requirement: Initial Portfolio Setup
사용자 SHALL 초기 포트폴리오 설정을 통해 첫 리밸런싱 이전의 상태를 기록할 수 있어야 한다.

#### Scenario: 초기 포트폴리오 설정 성공
- **WHEN** 사용자가 초기 현금, 주식수, 평단가를 입력한다
- **THEN** 시스템은 초기 상태를 데이터베이스에 저장한다
- **AND** 첫 번째 V값은 계산하지 않고 "INITIAL" 상태로 표시한다
- **AND** 사용자가 2주 후 첫 리밸런싱을 시작할 수 있음을 안내한다

#### Scenario: 초기 설정 없이 리밸런싱 시도
- **WHEN** 사용자가 초기 포트폴리오 설정 없이 밸류 계산을 실행한다
- **THEN** 시스템은 "먼저 초기 포트폴리오를 설정해주세요" 메시지를 표시한다
- **AND** 메인 메뉴로 돌아간다

#### Scenario: 초기 설정 데이터 검증
- **WHEN** 사용자가 음수의 현금이나 주식수를 입력한다
- **THEN** 시스템은 "현금과 주식수는 0 이상이어야 합니다" 오류를 표시한다
- **AND** 데이터를 저장하지 않는다

### Requirement: Bi-weekly Value Calculation
시스템 SHALL 2주마다 금요일 TQQQ 종가를 기반으로 새로운 V값(V2)을 계산할 수 있어야 한다.

#### Scenario: 첫 번째 V값 계산 (초기 리밸런싱)
- **GIVEN** 사용자가 초기 포트폴리오를 설정했다 (현금: pool, 주식수: shares)
- **AND** 가장 최근 금요일 TQQQ 종가를 조회했다 (price)
- **WHEN** 사용자가 밸류 계산을 실행한다
- **THEN** 시스템은 평가금액 E를 계산한다 (E = price × shares)
- **AND** v1 = E로 설정한다 (첫 번째 리밸런싱이므로)
- **AND** V2 = E + (pool/10) + ((E-E)/(2×√10))을 계산한다
- **AND** 계산 결과를 저장하고 표시한다

#### Scenario: 두 번째 이후 V값 계산
- **GIVEN** 이전 리밸런싱 기록이 존재한다 (v1 값 있음)
- **AND** 현재 현금: pool, 주식수: shares
- **AND** 가장 최근 금요일 TQQQ 종가를 조회했다 (price)
- **WHEN** 사용자가 밸류 계산을 실행한다
- **THEN** 시스템은 평가금액 E를 계산한다 (E = price × shares)
- **AND** V2 = v1 + (pool/10) + ((E-v1)/(2×√10))을 계산한다
- **AND** 계산 결과를 저장하고 표시한다
- **AND** v1을 다음 리밸런싱을 위해 저장한다

#### Scenario: 자동 가격 조회 성공
- **WHEN** 사용자가 밸류 계산을 실행한다
- **THEN** 시스템은 야후 파이낸스 API에서 TQQQ의 가장 최근 금요일 종가를 자동으로 가져온다
- **AND** 조회한 가격을 사용자에게 표시한다
- **AND** 확인 후 계산을 진행한다

#### Scenario: 자동 가격 조회 실패 시 수동 입력
- **WHEN** 야후 파이낸스 API 연결이 실패한다
- **THEN** 시스템은 "가격 조회에 실패했습니다. 수동으로 입력해주세요" 메시지를 표시한다
- **AND** 사용자가 금요일 TQQQ 종가를 직접 입력할 수 있게 한다

#### Scenario: 오프라인 모드에서 계산
- **WHEN** 인터넷 연결이 없다
- **THEN** 시스템은 캐싱된 마지막 가격을 사용할 것인지 묻는다
- **WHEN** 사용자가 동의한다
- **THEN** 캐싱된 가격으로 계산을 진행한다

#### Scenario: 밸류 계산 상세 표시
- **WHEN** V2 계산이 완료되면
- **THEN** 시스템은 다음 정보를 표시한다:
  - 평가금액 (E = 종가 × 주식수)
  - 이전 V값 (v1)
  - 현재 현금 (pool)
  - 계산된 새 V값 (V2)
  - V1 대비 변화율

### Requirement: Rebalancing Signal Detection
시스템 SHALL V2 값을 기준으로 ±15% 범위를 벗어나는지 판단하여 매수/매도/유지 신호를 제공해야 한다.

#### Scenario: 매수 신호 (V2가 상한선 초과)
- **GIVEN** V2 값이 계산되었다
- **AND** V2 > 이전 V값 × 1.15 (15% 초과)
- **WHEN** 계산 결과를 표시할 때
- **THEN** 시스템은 "매수 신호 (BUY)"를 녹색으로 표시한다
- **AND** "V2가 상한선(15%)을 초과했습니다" 메시지를 표시한다
- **AND** 현재 포트폴리오 비중 조정을 권장한다

#### Scenario: 매도 신호 (V2가 하한선 미만)
- **GIVEN** V2 값이 계산되었다
- **AND** V2 < 이전 V값 × 0.85 (15% 미만)
- **WHEN** 계산 결과를 표시할 때
- **THEN** 시스템은 "매도 신호 (SELL)"를 빨간색으로 표시한다
- **AND** "V2가 하한선(15%)을 하회했습니다" 메시지를 표시한다
- **AND** 일부 포지션 정리를 권장한다

#### Scenario: 유지 신호 (정상 범위)
- **GIVEN** V2 값이 계산되었다
- **AND** 이전 V값 × 0.85 ≤ V2 ≤ 이전 V값 × 1.15
- **WHEN** 계산 결과를 표시할 때
- **THEN** 시스템은 "유지 (HOLD)"를 노란색으로 표시한다
- **AND** "정상 범위 내입니다. 현재 전략 유지" 메시지를 표시한다

### Requirement: 5-Week Distributed Purchase Simulation
시스템 SHALL 5주 단위 분할 매수 시뮬레이션을 제공하여 다양한 가격 시나리오에서의 매수/매도 시점을 예측할 수 있어야 한다.

#### Scenario: 시뮬레이션 설정 및 실행
- **WHEN** 사용자가 5주 분할 매수 시뮬레이션을 선택한다
- **THEN** 시스템은 다음 정보를 입력받는다:
  - 총 투자 금액
  - 분할 횟수 (기본값: 5주)
  - 주간 가격 변동 폭 (기본값: ±10%, ±20%, ±30%)
- **AND** 각 시나리오별로 5주간 매수/매도 시점을 시뮬레이션한다

#### Scenario: 시뮬레이션 결과 표시 - 테이블 형식
- **WHEN** 시뮬레이션 계산이 완료되면
- **THEN** 시스템은 다음 형식으로 결과를 표시한다:
```
시나리오: +20% 상승 (주당 +20%)
┌──────┬────────┬────────┬──────────┬────────┐
│ 주차 │ 종가   │ 매수액 │ V값      │ 신호   │
├──────┼────────┼────────┼──────────┼────────┤
│ 1    │ $45.32 │ $2,000 │ 102,500  │ BUY    │
│ 2    │ $54.38 │ $2,000 │ 105,800  │ HOLD   │
│ 3    │ $65.26 │ $2,000 │ 108,200  │ SELL   │
│ 4    │ $78.31 │ $2,000 │ 106,500  │ BUY    │
│ 5    │ $93.97 │ $2,000 │ 110,300  │ HOLD   │
└──────┴────────┴────────┴──────────┴────────┘
```
- **AND** 색상으로 매수(녹색)/매도(빨간색)/유지(노란색)를 구분한다

#### Scenario: 시뮬레이션 결과 해석
- **WHEN** 시뮬레이션 결과를 표시할 때
- **THEN** 시스템은 각 시나리오별 다음 통계를 제공한다:
  - 최종 V값
  - 총 수익/손실
  - 매수/매도 횟수
  - 추천 전략

#### Scenario: 복수 시나리오 비교
- **GIVEN** 사용자가 ±10%, ±20%, ±30% 세 가지 시나리오를 선택했다
- **WHEN** 시뮬레이션이 완료되면
- **THEN** 시스템은 세 시나리오를 나란히 비교 표시한다
- **AND** 각 시나리오의 결과를 쉽게 비교할 수 있게 한다

#### Scenario: 시뮬레이션 데이터 없음
- **WHEN** 사용자가 시뮬레이션을 선택한다
- **AND** 초기 포트폴리오 설정이 되어 있지 않다
- **THEN** 시스템은 "먼저 초기 포트폴리오를 설정해주세요" 메시지를 표시한다

### Requirement: Record Management
사용자 SHALL 과거 리밸런싱 기록을 조회, 편집, 삭제할 수 있어야 한다.

#### Scenario: 전체 기록 조회
- **WHEN** 사용자가 기록 조회를 선택한다
- **THEN** 시스템은 모든 리밸런싱 기록을 최신순으로 표시한다
- **AND** 각 기록에 다음 정보를 포함한다:
  - 날짜
  - TQQQ 종가
  - 주식수
  - 현금
  - V값 (v1, V2)
  - 리밸런싱 신호 (BUY/SELL/HOLD)
- **AND** 신호별 색상으로 구분한다

#### Scenario: 기록 편집
- **WHEN** 사용자가 특정 기록을 선택하고 편집을 선택한다
- **THEN** 시스템은 해당 기록의 세부 정보를 표시하고 수정 가능하게 한다
- **AND** 사용자가 종가, 주식수, 현금을 수정할 수 있게 한다
- **AND** 수정 후 V값을 자동으로 재계산한다
- **AND** 변경사항을 저장한다

#### Scenario: 기록 삭제
- **WHEN** 사용자가 특정 기록을 선택하고 삭제를 선택한다
- **THEN** 시스템은 "정말 삭제하시겠습니까?" 확인 메시지를 표시한다
- **WHEN** 사용자가 확인한다
- **THEN** 해당 기록을 삭제한다
- **AND** 삭제 성공 메시지를 표시한다

#### Scenario: 기록 필터링
- **WHEN** 사용자가 기록 조회 시 필터 옵션을 선택한다
- **THEN** 시스템은 다음 필터를 제공한다:
  - 날짜 범위
  - 신호 유형 (BUY/SELL/HOLD만)
- **AND** 필터링된 결과만 표시한다

#### Scenario: 기록 없음
- **WHEN** 사용자가 기록 조회를 선택한다
- **AND** 저장된 기록이 없다
- **THEN** 시스템은 "리밸런싱 기록이 없습니다" 메시지를 표시한다

### Requirement: Console User Interface
시스템 SHALL 사용자 친화적인 콘솔 인터페이스를 제공해야 한다.

#### Scenario: 메인 메뉴 표시
- **WHEN** 프로그램이 시작된다
- **THEN** 시스템은 다음 옵션들을 표시한다:
  - 1. 초기 포트폴리오 설정
  - 2. 밸류 계산 (2주 리밸런싱)
  - 3. 5주 분할 매수 시뮬레이션
  - 4. 기록 조회
  - 5. 기록 편집
  - 6. TQQQ 가격 히스토리
  - 7. 데이터 초기화
  - 8. 종료

#### Scenario: 색상 출력 지원
- **WHEN** 프로그램이 시작된다
- **THEN** 시스템은 ANSI 색상 코드를 지원하도록 설정한다
- **AND** Windows 10+에서는 콘솔 핸들을 설정하여 색상을 활성화한다
- **AND** 매수 신호는 녹색(\033[92m), 매도 신호는 빨간색(\033[91m), 유지는 노란색(\033[93m)으로 표시한다

#### Scenario: 사용자 입력 안내
- **WHEN** 사용자가 각 메뉴를 선택한다
- **THEN** 시스템은 명확한 입력 프롬프트를 제공한다
- **AND** 필수 입력 필드를 표시한다
- **AND** 기본값을 제공한다 (예: 분할 횟수: 5)
- **AND** 입력 예시를 보여준다

### Requirement: Data Persistence
시스템 SHALL 모든 리밸런싱 기록을 CSV 파일에 영구 저장해야 한다.

#### Scenario: 데이터 자동 저장
- **WHEN** 사용자가 밸류 계산을 완료한다
- **THEN** 시스템은 즉시 CSV 파일에 데이터를 추가한다
- **AND** 파일이 없는 경우 새로 생성한다

#### Scenario: 데이터 로드
- **WHEN** 프로그램이 시작된다
- **THEN** 시스템은 기존 CSV 파일에서 데이터를 로드한다
- **AND** 파일이 없는 경우 빈 데이터로 시작한다

#### Scenario: CSV 데이터 구조
- **WHEN** 데이터를 저장할 때
- **THEN** 시스템은 다음 필드 구조를 사용한다:
```csv
date,price,shares,pool,v1,value2,signal,created_at
2026-03-14,45.32,100,50000,100000,105000,HOLD,2026-03-14T10:30:00
```
- **AND** date: 리밸런싱 날짜 (YYYY-MM-DD)
- **AND** price: TQQQ 종가
- **AND** shares: 주식수
- **AND** pool: 현금
- **AND** v1: 이전 V값
- **AND** value2: 계산된 새 V값
- **AND** signal: BUY/SELL/HOLD
- **AND** created_at: 레코드 생성 타임스탬프

### Requirement: Data Reset Functionality
시스템 SHALL 사용자가 모든 기록을 안전하게 초기화할 수 있어야 한다.

#### Scenario: 데이터 초기화 성공
- **WHEN** 사용자가 데이터 초기화를 선택한다
- **THEN** 시스템은 첫 번째 경고 메시지를 표시한다
- **AND** "모든 데이터가 영구적으로 삭제됩니다" 경고를 포함한다
- **AND** 현재 총 기록 수를 표시한다
- **AND** 사용자에게 계속 진행할 것인지 묻는다
- **WHEN** 사용자가 확인한다 (예: 'yes' 입력)
- **THEN** 시스템은 두 번째 확인을 요청한다
- **AND** "정말로 모든 데이터를 삭제하시겠습니까?" 메시지를 표시한다
- **WHEN** 사용자가 다시 확인한다
- **THEN** 시스템은 CSV 파일을 삭제한다
- **AND** 메모리 상태를 초기화한다
- **AND** 성공 메시지를 표시한다

#### Scenario: 데이터 초기화 - 첫 번째 확인에서 취소
- **WHEN** 사용자가 데이터 초기화를 선택한다
- **THEN** 시스템은 첫 번째 경고 메시지를 표시한다
- **WHEN** 사용자가 취소한다 (예: 'no' 입력)
- **THEN** 시스템은 "초기화가 취소되었습니다" 메시지를 표시한다
- **AND** 메인 메뉴로 돌아간다
- **AND** 데이터는 유지된다

#### Scenario: 데이터 초기화 - 빔 데이터베이스
- **WHEN** 사용자가 데이터 초기화를 선택한다
- **AND** 저장된 기록이 없다
- **THEN** 시스템은 "삭제할 데이터가 없습니다" 메시지를 표시한다
- **AND** 메인 메뉴로 돌아간다

### Requirement: Price Data Caching
시스템 SHALL 조회한 가격 데이터를 CSV 파일에 영구 저장하여 캐시처럼 사용해야 한다.

#### Scenario: 가격 데이터 CSV 자동 저장
- **WHEN** 시스템이 야후 파이낸스 API에서 가격을 성공적으로 가져온다
- **THEN** 가격, 날짜, 조회 시간을 price_cache.csv 파일에 저장한다
- **AND** 동일 날짜의 데이터가 이미 존재하면 업데이트한다
- **AND** CSV 구조: date,close_price,fetched_at

#### Scenario: 가격 데이터 CSV 조회
- **WHEN** 시스템이 특정 날짜의 가격이 필요하다
- **THEN** 먼저 price_cache.csv에서 해당 날짜의 데이터를 조회한다
- **AND** 데이터가 존재하면 API 호출 없이 사용한다
- **AND** 데이터가 없으면 API에서 조회하고 CSV에 저장한다

#### Scenario: API 실패 시 CSV 대체
- **WHEN** 야후 파이낸스 API 연결이 실패한다
- **AND** price_cache.csv에 최근 7일 이내의 데이터가 존재한다
- **THEN** 시스템은 "API 연결 실패. 캐시된 가격을 사용합니다" 메시지를 표시한다
- **AND** 캐시된 가격의 날짜를 표시한다
- **AND** 사용자가 동의하면 캐시된 가격으로 계산을 진행한다

#### Scenario: 캐시된 데이터 오래됨
- **WHEN** price_cache.csv의 최신 데이터가 7일 이상 되었다
- **THEN** 시스템은 "캐시된 데이터가 7일 이상 되었습니다" 경고를 표시한다
- **AND** "가능하면 인터넷 연결 후 최신 가격을 가져오세요" 메시지를 추가한다
- **AND** 사용자가 계속 진행할지 확인한다

#### Scenario: 가격 캐시 수동 관리
- **WHEN** 사용자가 가격 캐시 관리를 선택한다
- **THEN** 시스템은 다음 옵션을 제공한다:
  - 캐시된 가격 전체 보기
  - 특정 날짜 가격 수동 추가/수정
  - 특정 기간 가격 삭제
  - 전체 캐시 초기화

### Requirement: TQQQ Price History Viewer
사용자 SHALL 날짜별 TQQQ 종가 기록을 조회하고 확인할 수 있어야 한다.

#### Scenario: 가격 히스토리 전체 조회
- **WHEN** 사용자가 가격 히스토리 조회를 선택한다
- **THEN** 시스템은 price_cache.csv의 모든 가격 데이터를 최신순으로 표시한다
- **AND** 다음 정보를 포함한다:
  - 날짜
  - 종가
  - 등락률 (이전 영업일 대비)
  - 데이터 소스 (API/Manual)
- **AND** 상승은 녹색, 하락은 빨간색으로 표시한다

#### Scenario: 가격 히스토리 기간 필터링
- **WHEN** 사용자가 특정 기간을 선택한다
- **THEN** 시스템은 해당 기간의 가격 데이터만 표시한다
- **AND** 기간 선택 옵션을 제공한다:
  - 최근 1주일
  - 최근 1개월
  - 최근 3개월
  - 사용자 지정 기간

#### Scenario: 가격 히스토리 통계
- **WHEN** 사용자가 가격 히스토리 조회를 선택한다
- **THEN** 시스템은 기간별 통계를 표시한다:
  - 최고가 (Highest)
  - 최저가 (Lowest)
  - 평균가 (Average)
  - 표준편차 (Volatility)
  - 기간 수익률

#### Scenario: 가격 데이터 수동 입력
- **WHEN** 사용자가 가격 데이터 수동 입력을 선택한다
- **THEN** 날짜와 종가를 입력받는다
- **AND** 입력된 데이터를 price_cache.csv에 저장한다
- **AND** 데이터 소스를 "Manual"로 표시한다
- **AND** 동일 날짜의 데이터가 존재하면 덮어쓸지 묻는다

#### Scenario: 가격 데이터 편집
- **WHEN** 사용자가 특정 날짜의 가격을 선택하고 편집을 선택한다
- **THEN** 종가를 수정할 수 있게 한다
- **AND** 수정 후 저장 시 데이터 소스를 "Manual"로 변경한다

#### Scenario: 가격 데이터 삭제
- **WHEN** 사용자가 특정 날짜의 가격을 선택하고 삭제를 선택한다
- **THEN** "정말 삭제하시겠습니까?" 확인 메시지를 표시한다
- **WHEN** 사용자가 확인한다
- **THEN** 해당 날짜의 가격 데이터를 삭제한다

#### Scenario: 가격 히스토리 없음
- **WHEN** 사용자가 가격 히스토리 조회를 선택한다
- **AND** price_cache.csv에 데이터가 없다
- **THEN** 시스템은 "가격 데이터가 없습니다. API에서 가져오거나 수동으로 입력해주세요" 메시지를 표시한다
- **AND** API에서 가져오기 옵션을 제공한다

#### Scenario: 가격 데이터 내보내기
- **WHEN** 사용자가 가격 데이터 내보내기를 선택한다
- **THEN** price_cache.csv의 내용을 사용자가 지정한 파일로 복사한다
- **AND** Excel/CSV 형식을 지원한다
